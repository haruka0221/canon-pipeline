## Part 1：識別子監査 / Identifier Audit

### 1.1 目的 / Purpose

年ズレ監査（前回完了）に続く作業として、OL の editions データに含まれる識別子の付与率を定量化し、「HathiTrust 等の外部資源へ OL 経由で接続できるか」を判断するための基礎データを得ること。

Following the year-mismatch audit (completed previously), the goal was to quantify identifier coverage in OL editions data to assess feasibility of connecting to external resources such as HathiTrust.

### 1.2 方法 / Method

**対象 / Target:** `tmp/work_keys_year_audit.txt`（100 works、`derived/ol_works_final_population.tsv` からの抽出サンプル）

**スクリプト / Script:** `scripts/audit_identifiers.py`（本日新規作成）

**設計のポイント / Design notes:**

- `jq` による事前の JSON 構造確認で `oclc_number`（単数形）と `oclc_numbers`（複数形）の**両方**が存在することを確認し、両フィールドを別個にカウントした上で `oclc_any`（いずれかに存在）として集約した。
- `ocaid`（Internet Archive ID）が 89% の work に存在していたため、外部接続の将来候補として追加した。
- `languages` は `[{"key": "/languages/eng"}]` 形式のリストであることを確認。リスト長 > 0 で「あり」と判定。
- 年ズレ監査の知見（offset0 の先頭50件は近年版に偏る）を踏まえ、`size_top > 50` の28件について `offset50` も取得して比較した。

Prior `jq` inspection confirmed that both `oclc_number` (singular) and `oclc_numbers` (plural) exist as separate fields; both were counted individually and aggregated into `oclc_any`. `ocaid` (Internet Archive ID) was added as a future candidate for external linkage. The design also incorporated the lesson from the year-mismatch audit: offset0 (first 50 editions) tends to be biased toward recent reprints, so offset50 data was collected for the 28 works with `size_top > 50`.

**実行コマンド / Commands:**
```bash
python3 scripts/audit_identifiers.py
# → offset0 全100件を計測
# → tmp/missing_offset50.txt に offset50 未取得ファイルを出力
# curl ループで offset50 を取得（20件）
# → python3 scripts/audit_identifiers.py を再実行して比較計測
```

## Part 2：HathiTrust 接続試験 / HathiTrust Connection Test

### 2.1 目的 / Purpose

識別子監査で確認した OCLC・ISBN・LCCN・ocaid を用いて HathiTrust Bibliographic API への照合を試み、ヒット率の見込みと実装可否を判断する。

Using identifiers confirmed in the audit (OCLC, ISBN, LCCN, ocaid), test whether the HathiTrust Bibliographic API can be used for record matching, and assess expected hit rates.

### 2.2 HathiTrust API の仕様確認 / API Specification

**エンドポイント / Endpoint:**
```
https://catalog.hathitrust.org/api/volumes/brief/{id_type}/{id_value}.json
```

**対応する id_type:** `oclc`, `isbn`, `lccn`, `ia`（Internet Archive ID）

**ヒットなし時のレスポンス / Zero-hit response:**
```json
{"records": {}, "items": []}
```

**ヒットあり時のレスポンス構造 / Hit response structure:**
```json
{
  "records": {
    "RECORD_ID": {
      "recordURL": "...",
      "titles": [...],
      "isbns": [...],
      "oclcs": [...],
      "lccns": [...],
      "publishDates": [...]
    }
  },
  "items": [
    {
      "htid": "...",
      "itemURL": "...",
      "rightsCode": "...",
      "usRightsString": "...",
      "orig": "...",
      "fromRecord": "..."
    }
  ]
}
```

`rightsCode` / `usRightsString` に著作権状態が入る（例: `"ic"` = in-copyright, `"Limited (search-only)"`）。

`rightsCode` and `usRightsString` contain rights status (e.g., `"ic"` = in-copyright, `"Limited (search-only)"`).

### 2.4 結論 / Conclusion

**OL の editions（offset0）に含まれる識別子からの HathiTrust 直接接続は困難。**

**Direct connection from OL editions (offset0) identifiers to HathiTrust is not feasible with current data.**

原因の分析 / Root cause analysis:

OL の editions に登録されている OCLC・ISBN は、近代の再版・POD（オンデマンド印刷）版に付与されたものが中心である。HathiTrust が所蔵・デジタル化しているのは主に図書館が所蔵していた20世紀初頭以前の物理本であり、両者が参照している版の「時代層」が異なる。これは識別子の誤りではなく、OL と HathiTrust が互いに異なる版の集合をカバーしているという構造的なミスマッチである。

The OCLC and ISBN values recorded in OL editions predominantly belong to modern reprints and print-on-demand (POD) editions. HathiTrust primarily holds digitized copies of physical volumes from libraries, representing early 20th-century or earlier print runs. This is not a data error but a structural mismatch: OL and HathiTrust cover different strata of editions.

**具体的な証拠 / Supporting evidence:**

- OL472073W（Agatha Christie, 1923）の ocaid: `bwb_KU-770-351`（Better World Books = 中古本業者）、`murderonlinks0000agat_q5o8`（近年スキャン版）。どちらも HathiTrust 未収録。
- OCLC 番号は editions の先頭（近年版）に集中しており、HathiTrust が所蔵する古い版の番号とは一致しない。

- OL472073W ocaid values were `bwb_KU-770-351` (Better World Books, a secondhand bookseller) and `murderonlinks0000agat_q5o8` (a recent scan), neither in HathiTrust.
- OCLC numbers in OL editions are concentrated in recent reprints and do not correspond to the older volumes held by HathiTrust.

## Part 3：今後の方針 / Next Steps

### 3.1 採用する方向性 / Adopted direction

**Internet Archive（IA）直接接続への切り替え**

OL と IA は同じ識別子空間（ocaid）を共有しており、OL の editions に含まれる `ocaid` は IA のスキャン本に直接対応する。本母集団（1880–1950年の英語フィクション）は米国著作権法上パブリックドメインに入っているものが多く、IA では全文公開・ダウンロード可能なケースが多いと予想される。

**Switch to direct Internet Archive (IA) connection.**

OL and IA share the same identifier space (`ocaid`). The `ocaid` values in OL editions directly correspond to IA scanned volumes. Many works in this corpus (English fiction, 1880–1950) are in the public domain under US copyright law and are expected to be freely accessible via IA.

### 3.2 次の具体タスク / Concrete next tasks

1. **ocaid の品質確認**: `bwb_` プレフィックスは業者登録で IA スキャンではない可能性がある。プレフィックスのパターンを分類し、実際に IA でアクセスできる ocaid の割合を確認する。
2. **IA Metadata API での確認**: `https://archive.org/metadata/{ocaid}` で対象の `mediatype`、`language`、`date` を確認し、スキャン本かどうかを判定する。
3. **フルテキストアクセスの確認**: `ocaid` が取得可能かつ IA で全文公開されているものの割合を計測する（`access` フィールド等）。
4. **100サンプルでのヒット率計測**: 識別子監査と同じ100サンプルを使い、ocaid → IA でのアクセス可能率を定量化する。

1. **Audit ocaid quality**: The `bwb_` prefix likely indicates a bookseller-registered entry, not an IA scan. Classify ocaid prefix patterns and estimate the proportion that correspond to actual IA scanned texts.
2. **IA Metadata API check**: For each ocaid, query `https://archive.org/metadata/{ocaid}` to inspect `mediatype`, `language`, and `date` fields to confirm whether the item is a scanned book.
3. **Full-text access check**: Measure the proportion of ocaid values that are freely accessible full-text on IA.
4. **Hit-rate measurement on 100 samples**: Using the same 100-work sample, quantify the ocaid → IA access success rate.

- Only 14 OCLC numbers were tested against HathiTrust, which is not statistically sufficient. However, the all-zero result combined with structural analysis supports the conclusion that direct OCLC-based matching is infeasible.
